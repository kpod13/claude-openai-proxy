## Context

`DebugMiddleware` в `internal/proxy/debug.go` сейчас логирует только метод, путь, статус и длительность. Заголовки запроса и ответа не логируются, что ограничивает диагностику в verbose-режиме. Изменение затрагивает только `DebugMiddleware` — один файл, никаких архитектурных решений не требуется.

## Goals / Non-Goals

**Goals:**
- Логировать заголовки входящего запроса при DEBUG-уровне.
- Логировать заголовки ответа при DEBUG-уровне.
- Маскировать значение `Authorization` (`Bearer ***`, `Basic ***` и т.д.) чтобы токены не попадали в логи.

**Non-Goals:**
- Логирование тела запроса/ответа (содержит пользовательские данные).
- Фильтрация других «чувствительных» заголовков — только `Authorization`.
- Изменение поведения в INFO-режиме.

## Decisions

### 1. Заголовки как единый slog-атрибут

Заголовки передаются в `slog` как `slog.Any("headers", map[string][]string{...})`. Альтернатива — по одному атрибуту на заголовок — создаёт непредсказуемое число полей и засоряет structured-лог.

### 2. Маскировка только `Authorization`

Заменяем значение `Authorization` на `<тип> ***` (сохраняем схему: `Bearer`, `Basic` и т.д.). Это позволяет понять тип аутентификации, не раскрывая credentials. Остальные заголовки пишем as-is.

### 3. Заголовки ответа через `statusRecorder`

`statusRecorder` уже обёртывает `ResponseWriter`. Заголовки ответа доступны через `rec.Header()` после вызова `next.ServeHTTP` — дополнительных структур не нужно.

## Risks / Trade-offs

- [Объём логов] Каждый запрос добавляет N строк заголовков → Mitigation: логирование только при `--verbose`, который включается явно.
- [Утечка данных через другие заголовки] Cookie, кастомные токены могут попасть в лог → Mitigation: за пределами scope; `Authorization` — минимально необходимое требование безопасности.
